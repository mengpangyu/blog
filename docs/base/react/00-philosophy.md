# React 哲学

## 组件化

组件化的思维体现在组成部分的聚合，因此在文件结构上，组件化思维与后端项目常规 MVC 等分层思想差异很大，他的另一层意思其实表达的是：凡是属于我的部分，都应该跟我放在一起，对内高内聚、对外低耦合

## 封装思考

先基于多个场景，提炼出相同的部分与不同的部分，相同的部分封装在内部逻辑中，不同的部分通过参数/属性等方式传入

封装的核心应用，在于我们是否能够准确的区分出来共性与差异，例如对于函数来说，共性的逻辑我们封装到函数代码中，差异的内容，通过参数传入

在封装思维的驱动下，一部分数据被单独抽离出来，如果标签表达的是 UI，那么就可以把传统组件拆分为数据层和 UI 层

数据层与 UI 层在某种意义上是一一对应的关系，理解这一点，对未来探索最佳实践时有极大帮助

追求 UI 层和数据层的所见即所得，是组件化的最高评判标准

## 数据驱动视图

是现代流行框架的一种主流开发思维，他能够帮助我们将 UI 组件的变化，抽象为数据的变化，从而极大的简化开发流程

在数据驱动视图的开发思维下，我们不再去考虑如何获取 DOM 元素的引用来完成对应的逻辑，而是学会将 UI 行为抽象成为数据，并通过改变数据的方式完成交互需求，我们只需要把最多的精力放在数据如何发生变化即可，通常情况下，通过 http 现成请求服务端数据，通过 I/O 线程监听用户输入信息，通过定时器线程执行对应的逻辑等方式都会导致数据变化

掌握「数据驱动 UI」的核心哲学为：什么时候需要从 UI 抽离数据出来

> 在交互中变化的 UI，需要抽离出数据，并用 state 存放

例子：比如列表为空，请求一次接口之后，列表有了数据，那么列表的内容就是变化，抽离出来的数据就是一个数组

```tsx
function App() {
  const [list, setList] = useState([])

  return (
    <List data={list} renderItem={(item) => <Item ={item} />} />
  )
}
```

## 组件最小粒度原则

最小粒度原则实际上是封装思维的进阶运用，考究的是封装能力的火候

1. 明确什么时候要封装
   1. 首先当代码量过大，造成阅读困难时
   2. 其次当相同或者类似的代码逻辑，出现多次时
2. 人为约定的代码规模
   1. 每一个组件的代码行数尽可能的保持在一定的数量之内，可以约定 200/150 行，根据团队情况来定，一旦超出约定行数，就要对组件进行拆分
3. 分析相同/类似 UI 的出现场景
   1. 例如当类似的列表出现多次，只是数据不同，数据格式相同，就可以封装 List 组件，核心在于找出组件的相同之处和差异之处
4. 关注数据
   1. 当组件之间共享数据，就应该把数据存放在他们最近的父组件之中，反过来说就是**如果数据不存在共享，那么数据就应该属于对应组件私有**

## 开关思维

开关思维是数据驱动视图的进一步提炼，它能有效的帮我们找到最佳实践：及改变数据的方式越简单越好，最好只有一行代码

开关思维中，数据是电流，开关就是控制电流的接口，当我们把数据抽象好了之后，开关的接口如何提供，是我们要去思考的问题，这样的思考会引导我们创建更为简单的组件

开关思维是建立在数据驱动 UI 的基础之上，当 UI 操作变得复杂，我们使用数据驱动 UI 的思维来简化开发代码，随着项目规模和场景日益复杂，当操作数据也变得复杂时，就需要想到开关思维

> **使用一个数据，去驱动多少个数据，再用多个数据，去驱动 UI**

这就是开关思维的核心，也是自定义 Hooks 的终极奥义，开关思维能极大的减少前端开发的工作量
